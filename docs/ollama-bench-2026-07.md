# Ollama 로컬 모델 벤치마크 (2026-07-10)

게이트웨이(`https://llm.tan-kim.com/v1`) 경유로 서버에 설치된 chat 모델 6종을 실측했다.
목적: opencode 등 코딩 에이전트에 물릴 **기본 로컬 모델 선정**의 근거 확보.
(모델들은 registry 등록 없이 패스스루로 호출된다 — 등록 정책은 `.claude/CLAUDE.md` 참고.)

- 실행 방식: OpenAI 호환 `/v1/chat/completions`, 비스트리밍, `temperature=0.2`(게이트웨이 응답 캐시 회피), 모델별 워밍업 1회 후 순차 실행
- 라우팅 검증: 전 요청의 `x-llm-route` 헤더에서 `served=ollama:<태그>` 확인(폴백·바꿔치기 없음)
- 채점: 도구 호출은 tool_calls 구조 검사, 코드는 **Node로 실제 실행해 assert 통과 여부**, 형식·JSON은 파서 검사

## 결과 요약

3라운드 12개 태스크. 상위 5개 모델이 정확도 만점 동률이라 효율 지표로 최종 순위를 정했다.

| 순위 | 모델 | 정확도 | 응답 지연 합(2·3차) | 간결성* | 평균 tok/s | 콜드 로드 |
|------|------|--------|--------------------:|--------:|-----------:|----------:|
| **1** | **ornith:9b** | 12/12 | **19.8s** | **64 tok** | 32.7 | 8.8s |
| 2 | qwen3.5:9b | 12/12 | 22.1s | 163 tok | 34.7 | 3.3s |
| 3 | gemma4:e4b | 12/12 | 19.9s | 639 tok | **46.1** | 4.9s |
| 4 | qwen3:14b | 12/12 | 23.3s | 91 tok | 26.8 | **2.9s** |
| 5 | gemma4:12b | 12/12 | 26.3s | 435 tok | 26.4 | 6.5s |
| 6 | qwen2.5-coder:7b | 4/6 (1차 탈락) | — | 20 tok | 34.5 | 2.8s |

\* 간결성 = 1차 `tool_restraint` 태스크(짧은 답이 정답인 질문)의 completion 토큰 수. 에이전트 루프에서는
매 턴의 출력이 다음 턴 컨텍스트에 누적되므로, 같은 정답이면 짧을수록 지연·컨텍스트 소모가 작다.

### 선정 이유 — ornith:9b

- 정확도 만점 + **엔드투엔드 지연 최저** + **출력이 가장 간결**(gemma4:e4b 대비 1/10).
- gemma4:e4b는 tok/s는 1위지만 출력이 극단적으로 장황해(639 tok) 실제 태스크 지연은 비슷하고,
  에이전트 대화에서는 컨텍스트를 그만큼 잡아먹는다.
- qwen3.5:9b가 균형 잡힌 2위(전 태스크 만점, 로드 빠름) — ornith에 문제가 생기면 1순위 대체재.
- 약점: 콜드 로드 8.8s(최장). 기본 모델로 상주시키면 무관하고, 여러 모델을 번갈아 쓸 때만 체감된다.
- ornith는 thinking 계열 신모델 — 게이트웨이의 `/api/tags` capability 기반 think 억제가 정상 동작함을
  실측으로 확인(빈 content 없음, 지연 정상).

### qwen2.5-coder:7b 주의

'coder'라는 이름과 달리 에이전트 용도로 **비추천**:

- `tool_single` 실패 — 도구가 주어지고 도구가 필요한 질문인데 호출하지 않고 말로 답함
- `code_gen` 실패 — 생성한 parseVersion 함수가 테스트 불통과 (만점 5개 모델은 전부 통과한 문제)

단순 자동완성·비도구 용도로는 여전히 쓸 수 있다.

## 태스크 구성

| 라운드 | 태스크 | 검증 내용 |
|--------|--------|-----------|
| 1차 (6종 전체) | tool_single | 단일 도구 호출: 함수명·인자 정확성 |
| | tool_multiturn | 도구 결과를 받아 최종 답변에 반영 (재호출 금지) |
| | tool_restraint | 도구가 있어도 무관한 질문엔 호출하지 않음 |
| | code_gen | parseVersion 구현 → 실행 검증 3케이스 |
| | korean_format | 한국어 형식 제약(번호 목록 정확히 3개, 잉여 텍스트 금지) |
| | json_extract | JSON-only 출력 + 스키마(name/age/city) 값 정확성 |
| 2차 (만점 5종) | tool_chain | read_file→write_file 2단계 체이닝, 다른 키 보존 |
| | code_hard | topKFrequent(빈도+사전순 tie-break) → 실행 검증 |
| | needle_8k | ~5k 토큰 문서 속 사실 회수(장문 프롬프트 처리 겸측) |
| 3차 (만점 5종) | parallel_tools | 두 도시 날씨를 한 턴에 병렬 tool_calls(순차 완료는 0.5점) |
| | bugfix | `sort()` 사전순 정렬 버그 발견·수정 → 실행 검증 4케이스 |
| | lru_cache | LRU 캐시 클래스 구현 → 시맨틱 검증 6케이스 |

특기: 3차 `parallel_tools`에서 **5개 모델 전부 한 턴 병렬 호출 만점** — 로컬 9~14B급도
에이전트 하네스의 병렬 도구 패턴을 문제없이 소화한다.

## 재현 방법

벤치 스크립트(Node 20+, 의존성 0)는 세션 스크래치패드에서 작성·실행했다. 재현하려면
`bench.mjs`(1차)·`bench2.mjs`(2차)·`bench3.mjs`(3차) 패턴으로 게이트웨이에 요청하면 된다.
핵심 주의점:

- `temperature`를 0이 아닌 값으로 지정할 것 — 게이트웨이 응답 캐시(`CACHE_TTL_SECONDS`)는
  temperature 미지정/0 요청만 캐시하므로, 0으로 돌리면 재실행 시 캐시가 응답해 지연이 왜곡된다.
- 모델별 워밍업 1회로 콜드 로드를 분리 측정할 것 — 24GB RAM에서 모델 스왑 비용이 첫 태스크에 섞인다.
- `x-llm-route` 헤더로 실제 응답 모델을 확인할 것 — 명시 라우팅은 폴백하지 않지만 검증 차원.
